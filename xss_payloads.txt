SESSION 26: CLIENT-SIDE ATTACK PAYLOADS

1. REFLECTED XSS
* Payload Used: <script>alert('XSS')</script>
* Concept: Why does this attack disappear when you refresh the page?
Answer: Because the malicious script is only delivered as part of the immediate URL parameter (?q=<script>...). The server simply reflects it back once. It is not permanently saved anywhere in the database.

2. STORED XSS (Cookie Theft)
* Cookie Value Stolen: [admin_secret_99812_do_not_share]
* Payload Used: <script>alert(document.cookie)</script>

3. CROSS-SITE REQUEST FORGERY (CSRF)
* Malicious URL Crafted: [http://127.0.0.1:8081/transfer?to=Attacker&amount=5000]
* Concept: What is the primary defense against CSRF?
Answer: he primary defense against CSRF is the implementation of Anti-CSRF Tokens. These are unique, unpredictable values generated by the server for the user's session. They must be included in every state-changing request (like a fund transfer). If the server doesn't see the correct token, it rejects the request, stopping  an attacker from "forging" an action on the user's behalf.
